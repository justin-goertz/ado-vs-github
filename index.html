01De aanleiding

De vraag is veranderd

Veel teams zijn op Azure DevOps gebleven omdat "Microsoft stack" een veilige keuze leek. Toen kwam AI. En toen werd de vraag: welk model gebruiken we?

Dat is de verkeerde vraag. De juiste vraag is: welk platform geeft me dat AI-model én alle andere dingen die ik er omheen nodig heb — zonder dat ik ze zelf moet bouwen?

Even eerlijk over Claude vs Copilot

Claude Code is een goede CLI-tool. Devs die er los mee werken zijn er vaak blij mee — terecht.

Twee dingen zijn belangrijk om te weten:

  • Claude zit ook in Copilot. Claude Sonnet 4.5, Opus 4 en Haiku zijn standaard model-opties in Copilot Chat, in de IDE, op github.com en in de CLI. Dezelfde modellen, alleen met een andere wrapper eromheen. Volledige modellenlijst.
  • Het verschil zit niet in het model, maar in de context. Claude Code werkt op één laptop met één repo open. Copilot werkt op platform-niveau: het ziet je PRs, issues, Actions, security alerts, dependency graph en je hele org. Een agent kan een issue oppakken, een branch maken, een PR openen, CCR-feedback verwerken, security-fixes toepassen en deployen — zonder dat een dev erbij hoeft.

Voor een individuele dev: prima keuze allebei. Voor een engineering organisatie van 20+ devs: het verschil is dat Copilot de hele pijplijn raakt — niet alleen het typen van code.

Met GitHub + Copilot ship je significant meer met dezelfde mensen

Dit gaat niet over modellen of features. Dit gaat over hoeveel meer code je team af krijgt per week, en hoeveel meer product je organisatie kan leveren met dezelfde headcount. De cijfers komen uit peer-reviewed onderzoek en grootschalige enterprise-studies.

Waarom dat ertoe doet voor jouw organisatie

Dit gaat niet over "GitHub is groter dan ADO". Het gaat over vier hele praktische gevolgen:

  • Elke nieuwe AI-tool, MCP-server of dev-integratie verschijnt eerst op GitHub. MCP servers, Marketplace apps, Actions — daar is het ecosysteem. Op ADO loop je structureel achter.
  • Je hiring pool werkt al op GitHub. Elke developer die je interviewt heeft een GitHub-profiel, niet een ADO-profiel. Onboarding is sneller, externe contributors plug-and-play.
  • Je security wordt beter doordat de wereld erop kijkt. 90M+ PRs per jaar betekent dat CVE's, supply-chain attacks en vulnerabilities hier eerst worden gevonden — en de fixes ook eerst hier landen via Dependabot en Code Scanning.
  • Je open source dependencies leven hier. Of je het wil of niet, je code hangt aan duizenden npm/PyPI/Maven packages — die allemaal op GitHub ontwikkeld worden. Op GitHub zit je in hetzelfde systeem als je hele dependency tree.

De gap is niet stilstand vs vooruitgang — het is achteruitgang in relatieve termen. Concurrenten die nu op GitHub + Copilot zitten, leveren meer software per kwartaal met hetzelfde budget. Elk jaar dat je op ADO blijft, groeit dat verschil.

03Het Windows / .NET verhaal

"Maar onze legacy producten…"

Een veelgehoord argument om op ADO te blijven: "Onze legacy producten draaien op Windows, daar zit ADO dichter op." Dat klopte. Het klopt niet meer.

  • GitHub-hosted Windows runners draaien Windows Server 2019, 2022 en 2025 met de volledige .NET stack (Framework + Core), MSBuild, Visual Studio Build Tools, IIS-tooling, en PowerShell — out of the box.
  • Self-hosted runners draaien op je eigen Windows boxen voor wat niet naar SaaS kan.
  • GitHub Copilot + de .NET Upgrade Assistant zijn expliciet gebouwd om .NET Framework → modern .NET migraties te versnellen.
  • De Visual Studio integratie van GitHub is first-class — dezelfde Microsoft-engineers bouwen het.

Legacy is geen reden meer om te blijven. Het is een argument om over te stappen mét hulp.

04De business case begint hier

Eén feature dekt de hele licentie

Je hoeft het hele platform niet eens te activeren om de business case rond te krijgen. Pak één capability: Copilot Code Review (CCR).

CCR reviewt elke PR automatisch vóór een mens er naar kijkt. Style, bugs, security smells, performance — opgevangen voordat het in de review queue ligt.

30–50%
minder review-tijd per PR (afhankelijk van team)
↑ velocity
meer PRs per week, dezelfde mensen

Reken een team van 20 engineers × gemiddeld 4u code review per week × NL senior rate. Op jaarbasis enkele tonnen aan review-tijd. CCR vangt een significant deel daarvan op.

Het volledige plaatje

Alleen al de tijdwinst en de snellere merge cadence van CCR dekt in veel gevallen de complete GitHub Enterprise + Copilot licentie.

Voor de rest van wat je krijgt — agents, security AI, codespaces, autofix, hele pijplijn van issue → PR → merge → deploy — betaal je effectief niets extra.

En dan ship je nog meer product met dezelfde mensen, op een veiliger platform, terwijl je stopt met zelf infra-tooling bouwen.

Laten we het samen uitrekenen

De getallen hierboven zijn algemeen. De échte business case wordt scherp als we het op jouw team toepassen: aantal engineers, gemiddelde PR-volume, review-tijd, fully-loaded cost per FTE, gemiddelde tijd-tot-merge, productie-incidenten per kwartaal.

Een half uur samen en we hebben een onderbouwd cijfer dat je intern kunt verdedigen — inclusief conservatieve en optimistische scenario's. Geen aannames, jullie context.

05Het rekensommetje

"We bouwen het zelf"

Stel je hebt al 2 engineers, 1 maand bezig met één pipeline agent zelf bouwen. Dat is een concreet getal. Hou dat vast.

Vermenigvuldig dat naar SAST, SCA, secret scanning, AI code review, security autofix, copilot metrics, audit streaming, etc. Snel zit je op een paar honderdduizend euro engineering tijd — alleen al om op niveau te komen van wat een GitHub seat per maand kost.

De verborgen kostenposten

  • Maintenance forever. Wat je bouwt, moet je onderhouden. CodeQL queries, Dependabot ecosystem coverage, secret scanning patterns — dat zijn full-time research teams bij GitHub.
  • Opportunity cost. Die 2 FTE bouwen geen product features.
  • Talent. Senior devs solliciteren niet om interne CI plumbing te bouwen. Ze willen op een modern platform werken.
  • Je komt nooit op niveau van native AI op security findings — zie sectie 06.

07EU compliance

AVG, NIS2, DORA — een apart hoofdstuk

Voor Nederlandse organisaties met AVG, NIS2 of DORA in scope is dit geen optie, het is een eis.

  • GitHub Enterprise Cloud met Data Residency (GHE.com) — code, metadata en audit logs blijven binnen de EU. Eigen subdomain (*.ghe.com), aparte tenancy, EU regio.
  • Copilot Data Residency in EU — prompts en completions verwerkt binnen EU regio.
  • SOC 2 Type II, ISO 27001, FedRAMP Moderate — full compliance stack.

ADO heeft EU hosting, maar geen aparte EU-tenancy variant met dezelfde data residency garanties als GHE.com.

08Het gat dat niet dicht gaat

De ene gap die je nooit invult

⚠ Native AI op security findings

Sommige gaps zijn op te lossen met budget en tijd. Native AI op security findings niet.

Dat werkt alleen als je CodeQL (SAST) en Dependabot (SCA) als platform-basis hebt. Daar is Copilot Autofix bovenop gebouwd.

Zelfs als je security tooling van derde partijen koopt en eraan vast plakt, krijg je nooit dezelfde geïntegreerde flow: vinding → AI-suggestie → autofix → PR → merge, binnen één platform met één audit trail.

09Wat Microsoft zelf doet

Dit is wat Microsoft zelf schrijft. Niet wat GitHub erover zegt.

Microsoft bezit zowel Azure DevOps als GitHub. Voor de richting van het platform hoef je niemand van GitHub te geloven — Microsoft is er zelf opvallend expliciet over. Hieronder de publieke bronnen, met directe quotes.

1. ADO staat officieel op "security en onderhoud"

  • De Azure DevOps features timeline bevat geen nieuwe feature-investeringen meer — de roadmap focust op security, compliance en stabiliteit.
  • In de Sprint 254 release notes kondigt Microsoft de uitfasering van het Azure DevOps OAuth-platform aan, woordelijk: "a first step towards retiring the Azure DevOps OAuth platform". Klanten worden doorverwezen naar Microsoft Entra ID — dezelfde identity layer als GitHub Enterprise.

2. Microsoft zegt zelf: migreer nu naar GitHub

  • Het Azure DevOps team schrijft op de officiële DevOps blog: "Now is the time to migrate your repos to GitHub" — Azure DevOps with GitHub repositories: your path to agentic AI.
  • In de follow-up (nov 2025) positioneert Microsoft GitHub als "the richest end-to-end agentic experience" — Next steps in the path to agentic AI.
  • Microsoft verhuist z'n eigen interne repos van ADO naar GitHub, en publiceert het draaiboek: How Microsoft is migrating repositories to GitHub.

3. Nadella's CoreAI-memo zet de richting vast

  • In januari 2025 kondigt Satya Nadella de nieuwe CoreAI – Platform and Tools divisie aan. GitHub valt daar expliciet onder, samen met Dev Div en de AI Platform-teams. ADO wordt niet genoemd als groeigebied.
  • Dit is geen marketing — dit is een org-structuur. Waar het geld en de engineers heen gaan, gaat de roadmap heen.

4. Microsoft eigen engineering zit al op GitHub

  • github.com/microsoft — VS Code, TypeScript, PowerToys, WSL, Playwright, .NET, Azure SDK's.
  • github.com/dotnet, github.com/Azure, github.com/openai — alles wat Microsoft als platform pusht, ontwikkelen ze in de open op GitHub. Niet op ADO.

5. De migratiepaden zijn first-party gebouwd

  • GitHub Enterprise Importer — ADO repos → GitHub, door Microsoft onderhouden.
  • GitHub Actions Importer — Azure Pipelines YAML → GitHub Actions, automatisch.
  • Een vendor bouwt geen first-party tools van zijn ene product naar zijn andere tenzij dat de gewenste richting is.
  • GitHub publiceert er een hele resource-pagina overheen: Accelerate innovation with an Azure DevOps migration — met als sluitende boodschap "the answer is clear: now is the moment to migrate".

Dit is geen "ADO is dood" verhaal — ADO blijft draaien voor wie er op zit. Maar als Microsoft zelf schrijft "now is the time to migrate your repos to GitHub", z'n eigen repos verhuist, en GitHub als "richest end-to-end agentic experience" positioneert, dan is dat een signaal dat je serieus tegen je eigen 3–5 jaars-strategie kan houden.
