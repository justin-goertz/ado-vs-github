Korte vergelijking voor engineering leaders

Het is niet Claude vs Copilot.
Het is Azure DevOps vs GitHub.

De AI-modelkeuze is de makkelijke. De platformkeuze daaronder bepaalt wat je de komende 5 jaar wel of niet hoeft te bouwen.

De aanleiding

Veel teams zijn op Azure DevOps gebleven omdat "Microsoft stack" een veilige keuze leek. Toen kwam AI. En toen werd de vraag: welk model gebruiken we?

Dat is de verkeerde vraag. De juiste vraag is: welk platform geeft me dat AI-model én alle andere dingen die ik er omheen nodig heb — zonder dat ik ze zelf moet bouwen?

Eerst even: het Windows / .NET Framework verhaal

Een veelgehoord argument om op ADO te blijven: "Onze legacy producten draaien op Windows, daar zit ADO dichter op." Dat klopte. Het klopt niet meer.

  • GitHub-hosted Windows runners draaien Windows Server 2019, 2022 en 2025 met de volledige .NET stack (Framework + Core), MSBuild, Visual Studio Build Tools, IIS-tooling, en PowerShell — out of the box.
  • Self-hosted runners draaien op je eigen Windows boxen voor wat niet naar SaaS kan.
  • GitHub Copilot + de .NET Upgrade Assistant zijn expliciet gebouwd om .NET Framework → modern .NET migraties te versnellen.
  • De Visual Studio integratie van GitHub is first-class — dezelfde Microsoft-engineers bouwen het.

Legacy is geen reden meer om te blijven. Het is een argument om over te stappen mét hulp.

Het rekensommetje achter "we bouwen het zelf"

Stel je hebt al 2 engineers, 1 maand bezig met één pipeline agent zelf bouwen. Dat is een concreet getal. Hou dat vast.

~€25–35k 2 FTE × 1 maand all-in (NL senior dev rates)
1 van ~20 capabilities die GitHub native levert — die je dus zou moeten bouwen

Vermenigvuldig dat naar SAST, SCA, secret scanning, AI code review, security autofix, copilot metrics, audit streaming, etc. Snel zit je op een paar honderdduizend euro engineering tijd — alleen al om op niveau te komen van wat een GitHub seat per maand kost.

De verborgen kostenposten

  • Maintenance forever. Wat je bouwt, moet je onderhouden. CodeQL queries, Dependabot ecosystem coverage, secret scanning patterns — dat zijn full-time research teams bij GitHub.
  • Opportunity cost. Die 2 FTE bouwen geen product features.
  • Talent. Senior devs solliciteren niet om interne CI plumbing te bouwen. Ze willen op een modern platform werken.
  • Je komt nooit op niveau van native AI op security findings — zie sectie 06.

EU compliance — een apart hoofdstuk

Voor Nederlandse organisaties met AVG, NIS2 of DORA in scope is dit geen optie, het is een eis.

  • GitHub Enterprise Cloud met Data Residency (GHE.com) — code, metadata, en audit logs blijven binnen de EU. Eigen subdomain (*.ghe.com), aparte tenancy, EU regio.
  • Copilot Data Residency in EU — prompts en completions verwerkt binnen EU regio.
  • SOC 2 Type II, ISO 27001, FedRAMP Moderate — full compliance stack.

ADO heeft EU hosting, maar geen aparte EU-tenancy variant met dezelfde data residency garanties als GHE.com.

Het gat dat je nooit dicht krijgt

Sommige gaps zijn op te lossen met budget en tijd. Native AI op security findings niet.

Dat werkt alleen als je CodeQL (SAST) en Dependabot (SCA) als platform-basis hebt. Daar is Copilot Autofix bovenop gebouwd.

Zelfs als je security tooling van derde partijen koopt en eraan vast plakt, krijg je nooit dezelfde geïntegreerde flow: vinding → AI-suggestie → autofix → PR → merge, binnen één platform met één audit trail.

De signalen die Microsoft zelf geeft

Microsoft bezit zowel ADO als GitHub. Kijk naar waar de R&D investering en de roadmap energie heen gaan:

  • De Azure DevOps release notes tonen vooral kleine verbeteringen en bugfixes. Geen nieuwe AI capabilities, geen nieuwe agents.
  • De GitHub changelog publiceert wekelijks nieuwe features — agents, modellen, security AI, MCP support.
  • Microsoft heeft expliciet een GitHub Enterprise Importer en een Actions Importer voor ADO Pipelines gebouwd. Een vendor maakt geen migratietools naar zijn andere product tenzij dat de gewenste richting is.
  • De Azure DevOps blog verwijst zelf regelmatig naar GitHub voor nieuwe AI scenarios.
  • Microsoft engineering teams (Azure, Windows, Office) hosten hun open source op github.com/microsoft, niet op dev.azure.com.

Dit is geen "ADO is dood" verhaal. ADO blijft draaien voor wie er op zit. Maar nieuwe investering — en zeker AI investering — gaat naar GitHub.

Bonus: het Microsoft EA voordeel

Voor organisaties met een bestaande Microsoft Enterprise Agreement: GitHub Enterprise en Copilot zijn beschikbaar via dezelfde MS EA. Geen aparte vendor, geen aparte PO flow, gebruik je commit. Praktisch betekent dit dat procurement een non-issue is.

De juiste vraag

Niet: "Wat is goedkoper, Claude direct of Copilot?"

Wel: "Op welk platform bouwen we de komende 5 jaar — en hoeveel zelf-bouw stoppen we erin om datzelfde platform niveau te halen?"

Volgende stap

Als dit verhaal raakt aan iets dat bij jou speelt — een ADO migratie die op pauze staat, een team dat custom agents gaat bouwen, of een AI strategie waar het platform onder ontbreekt — neem dan contact op met je GitHub team.

De vraag is niet of je naar GitHub gaat. De vraag is wanneer, en hoeveel je tot dan toe zelf hebt gebouwd dat je daarna weer weggooit.
